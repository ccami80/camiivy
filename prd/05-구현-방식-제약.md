# CAMI & IVY — 구현 방식 제약

**모든 개발은 아래 제약을 준수한다. 예외 없음.**

---

## 기술·구조

| 항목 | 제약 |
|------|------|
| **언어** | **JavaScript 기준으로만 작성** (TypeScript 사용 금지) |
| **라우팅** | **Next.js App Router 구조 유지** |
| **API** | Next.js API Routes 사용 |
| **DB** | **MySQL 기준 스키마 설계** (운영·배포 시 MySQL) |
| **ORM** | **Prisma ORM을 사용한 실제 DB 연동** |

---

## 동작 기준

- **관리자 페이지와 API Routes를 함께 구현**한다. (관리자 화면 추가 시 해당 API도 반드시 구현)
- **페이지와 API를 함께 구현**한다. (페이지만 있거나 API만 있는 구현 금지)
- **UI만 있는 페이지**는 허용하지 않는다.
- **mock 데이터만 있는 구현**은 허용하지 않는다.
- **실제 동작 가능한 CRUD**만 구현한다. (조회·생성·수정·삭제가 Prisma로 실제 DB와 연동되어야 함)

---

## 체크리스트 (기능 추가 시)

- [ ] 해당 기능의 **API Route**가 구현되어 있는가?
- [ ] API가 **Prisma**로 MySQL(SQLite 로컬 포함)에 실제 **CRUD**를 수행하는가?
- [ ] **페이지**에서 해당 API를 호출해 **실제 데이터**를 보여주고, 폼 제출 시 API가 호출되는가?
- [ ] 가짜 배열/객체만 렌더링하는 **정적·목업 전용 화면**이 아닌가?
